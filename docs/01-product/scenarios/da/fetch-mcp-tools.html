VS Code Fetch Action Flow

The built-in Start CodeLens runs the MCP server; the ATK CodeLens then triggers the entire fetch-and-update flow.

1. Start the MCP server

VS Code's MCP runtime owns Start and More...; ATK only contributes the leading ⚡ ATK: Fetch action from MCP action. Tool and prompt counts are not shown until the server is running.

2. Click ⚡ ATK: Fetch action from MCP

Once the server is running, VS Code populates the tool and prompt counts. Clicking the CodeLens (or ATK: Update Action with MCP from the Command Palette) discovers the server's tools and then drives the manifest/op/auth pickers below.

3. Select the action manifest you want to update

4. Name the new action manifest file only when "Create a new" picked

5. Select Operation(s) Copilot can interact with

The selected operations are written into the action manifest picked in step 3 (here appPackage/ai-plugin.json) as entries under functions and referenced from runtimes[].run_for_functions.

6. Select Authentication Type only when MCP server requires auth

7. Success notification

Recoverable: tools not found blocks step 3

User-visible outputs

Mirrors fetch-mcp-tools.md § User-visible outputs. This scenario updates an existing DA project; there is no template boilerplate to summarize. Each output below is driven by the picker answers in the flow above.

File changes

  • appPackage/<chosen action manifest>.json — created when step 3 picks Create a new ai-plugin.json (name from step 4), modified otherwise. Operations from step 5 are written under functions and referenced from the MCP server runtime's run_for_functions; the runtime's auth block is written when step 6 selects an authentication type.
  • appPackage/declarativeAgent.json — modified only when a new manifest was created in step 3. A new entry is appended to actions with a non-colliding id (action, action_2, ...) and file set to the new manifest's basename.
  • m365agents.yml — modified only when an authentication type is selected and the matching registration step is not already present. The injected step shape and ordering match SCN-DA-CREATE-WITH-MCP-SERVER and SCN-DA-ADD-MCP-ACTION-TO-DA.

Notifications

Source: Microsoft 365 Agents Toolkit. The Provision button runs the standard provision lifecycle.

Error and recovery messages

Surfaced when the MCP server returns no tools at discovery time. Recovery: start the server using the built-in VS Code MCP Start CodeLens (or fix its configuration) and re-click ⚡ ATK: Fetch action from MCP. No files are written before this fires.

Prerequisite errors fire when .vscode/mcp.json is missing, malformed, has no server entries, or the resolved server entry is incomplete (no url for http, no command for stdio). Each toast names the missing piece; the user edits the file and retries. No files are written before these fire.

Cancellation at any picker leaves no notification and writes nothing to disk.

Environment and secret writes

The OAuth registration step injected into m365agents.yml reserves a configuration id that is written into env/.env.* when the provision lifecycle runs. This scenario does not write to env/.env.* at picker time; auth secret collection in VS Code follows the convention defined in SCN-DA-CREATE-WITH-MCP-SERVER and SCN-DA-ADD-MCP-ACTION-TO-DA.

External side effects

None at scenario time. Cloud-side OAuth client registration happens later during the provision lifecycle, when the injected oauth/register (or dcr/register) step runs.
