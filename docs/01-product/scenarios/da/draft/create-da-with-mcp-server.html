Visualizes the redesigned VS Code create flow from create-da-with-mcp-server.md (draft). In this flow, the authentication-type pick and its conditional follow-up fields are part of the create flow itself. No static MCP tools file is captured; tool discovery happens at runtime. Feature flag

This scenario is gated behind a new feature flag while the design lands — default off so that current shipping behavior is unchanged.

TEAMSFX_MCP_FOR_DA_DT default: false

Gates the create-flow changes documented on this page: (1) the scaffold persists the OAuth client id / scope / secret answers under project-standard UPPER_SNAKE names in env/.env.<env> and env/.env.<env>.user; (2) the oauth/register step in m365agents.yml references those entries via ${{...}}. When the flag is off, current shipping behavior is preserved: env files contain only the auth-id placeholder, the YAML omits the credential trio, and the provision step picks up credentials through the in-process bridge populated by the create flow.

MCP-backed DA itself is now unconditionally available in the create flow (the old umbrella flag has been removed). This new flag scopes only the credential-persistence subset described above and is intended to be temporary — removed once the rollout is complete.

TEAMSFX_MCP_FOR_DA_DCR default: false

Gates only the OAuth (with dynamic registration) auth-type option in step 6 and the matching dcr/register action in m365agents.yml. The option is shown — and the CLI value oauth-dynamic is accepted — only when both TEAMSFX_MCP_FOR_DA_DT and TEAMSFX_MCP_FOR_DA_DCR are true, because DCR scaffolding currently piggybacks on the DT-mode env-file plumbing. Default off; temporary — removed once DCR ships unconditionally.

VS Code Create Flow

Guided prompts before the DA project exists. The generated project already has the MCP action wired into the manifest and the m365agents.yml; no follow-up fetch step is required.

1. Choose project type

2. Choose DA template path

3. Choose action source

4. Choose MCP server type (auto-skipped when only Remote is available; branches into steps 5–6 for Remote, step 5L for Local)

5. Enter MCP server URL Remote branch only

Recoverable: missing MCP server URL Remote branch only · blocks step 6

6. Select authentication type Remote branch only · new

Note: the OAuth (with dynamic registration) option is only shown when both TEAMSFX_MCP_FOR_DA_DT and TEAMSFX_MCP_FOR_DA_DCR are true (see Feature flag section). With either flag off, the pick has three options.

VS Code gate: the whole pick (and follow-up cards 6a/6b) is only shown when TEAMSFX_MCP_FOR_DA_DT is true. When the flag is off, the create flow skips straight from the MCP server URL to the project location step — matching shipped behavior; auth-type is collected later by the fetch-mcp-tools CodeLens (SCN-DA-FETCH-MCP-TOOLS). CLI is unaffected.

6a. OAuth (static) follow-up Remote branch only · only when option above is selected

6b. Entra SSO follow-up Remote branch only · only when Entra SSO is selected

5L. Select a local MCP server Local branch only · replaces steps 5–6 for Local

Pre-discovered via odr.exe; no URL, no authentication-type pick, and no follow-up fields. The chosen server's command/args are written straight into .vscode/mcp.json. m365agents.yml gets no oauth/register or dcr/register step.

7. Choose project location

8. Enter app name

9. Project ready to provision and run

Note: the scaffold does not auto-open .vscode/mcp.json after creation. The notification surfaces only the primary Provision action; there is intentionally no “Fetch action from MCP” button, since that path is a fallback for when dynamic tool discovery fails rather than the recommended default.

9w. Recoverable: DCR well-known URL not auto-discovered OAuth (dynamic) only

When the auth-type is OAuth (with dynamic registration), the scaffold probes the MCP server (HTTP POST with an MCP initialize JSON-RPC body) to discover the OAuth well-known authorization-server URL via the WWW-Authenticate resource_metadata chain. If discovery fails — e.g. the server returns 401 without resource_metadata, or the probe times out — the dcr/register step is still injected, but wellKnownAuthorizationServer is set to the literal string <PLEASE_FILL_IN_WELL_KNOWN_AUTHORIZATION_SERVER_URL> and a warning notification is surfaced. The developer must replace the placeholder before provisioning.

Files written after success

Shown for an example DA named github-da against https://api.githubcopilot.com/mcp/, with authentication type OAuth (with static registration). The scaffolded template boilerplate is not shown here; only the files whose content is driven by the developer's answers are listed. The oauth/register step is injected by ActionInjector.injectCreateOAuthActionForMCP in m365agents.yml for OAuth (static) and Entra SSO; the static variant adds authorizationUrl/tokenUrl/refreshUrl with identityProvider: Custom, the SSO variant uses identityProvider: MicrosoftEntra only. OAuth (dynamic) instead injects a dcr/register step (DCR — Dynamic Client Registration, RFC 7591). None writes no auth step. .vscode/mcp.json is written for every authentication type.

appPackage/ai-plugin.json generated, dynamic tool discovery

The runtime uses type: RemoteMCPServer with enable_dynamic_discovery: true and run_for_functions: ["*"], so tool definitions are fetched from the MCP server at request time rather than being baked into functions. auth is shown for OAuth (static); for None the entry becomes { "type": "None" } and the auth block is omitted.

appPackage/ai-plugin.json
1{
2"$schema": "https://developer.microsoft.com/json-schemas/copilot/plugin/v2.4/schema.json",
3"schema_version": "v2.4",
4"name_for_human": "github-da",
5"description_for_human": "github-da${{APP_NAME_SUFFIX}}",
6"namespace": "github_da",
7"contact_email": "publisher-email@example.com",
8"functions": [],
9"runtimes": [
10{
11"type": "RemoteMCPServer",
12"spec": { "url": "https://api.githubcopilot.com/mcp/", "enable_dynamic_discovery": true },
13"run_for_functions": [ "*" ],
14"auth": { "type": "OAuthPluginVault", "reference_id": "${{MCP_DA_AUTH_ID_APIGITHUBC}}" }
15}
16]

appPackage/declarativeAgent.json generated, references action

The action id is the template default action_1. It identifies the action within this manifest and is not surfaced to provisioning — the OAuth / DCR step injected into m365agents.yml below uses the plugin namespace (here github_da) as its step name: instead.

appPackage/declarativeAgent.json

2"$schema": "https://developer.microsoft.com/json-schemas/copilot/declarative-agent/v1.7/schema.json",
3"version": "v1.7",
4"name": "github-da",
5"actions": [
6{ "id": "action_1", "file": "ai-plugin.json" }
7]
8}

m365agents.yml + auth registration step

Injected immediately after the existing teamsApp/create step. Per auth type:

  • OAuth (static) — oauth/register with identityProvider: Custom plus authorizationUrl / tokenUrl / refreshUrl, and clientId / clientSecret / scope as ${{...}} references to the UPPER_SNAKE env entries the scaffold persisted. Injected by ActionInjector.injectCreateOAuthActionForMCP.
  • Entra SSO — same oauth/register shape with identityProvider: MicrosoftEntra, no URL trio, and only clientId: ${{MCP_DA_OAUTH_CLIENT_ID_APIGITHUBC}} (driver suppresses secret / scope / confirm). Env-var suffix is derived from the MCP server hostname (here api.githubcopilot.com → APIGITHUBC); multiple servers get distinct suffixes.
  • OAuth (dynamic) — dcr/register (RFC 7591 Dynamic Client Registration) with name: github_da, appId: ${{TEAMS_APP_ID}}, applicableToApps: SpecificApp, targetAudience: HomeTenant, wellKnownAuthorizationServer: <RFC 8414 metadata URL>, targetUrlsShouldStartWith: [ <mcp url> ], and the same writeToEnvironmentFile.configurationId: MCP_DA_AUTH_ID_APIGITHUBC. No client-credentials prompts — the authorization server issues them dynamically at provision. The yml schema version: is also bumped to v1.13 on injection (other variants stay at the template default v1.12) because dcr/register requires schema v1.13+. If the well-known URL can’t be auto-discovered, the value is written as the placeholder <PLEASE_FILL_IN_WELL_KNOWN_AUTHORIZATION_SERVER_URL> and a warning notification is surfaced — see 9w above.
  • None — no registration step.

env/.env.dev scaffold persists OAuth answers; configurationId placeholder filled at provision

Scaffolded by the declarative-agent-with-action-from-mcp template when McpAuth is set. The scaffold writes the OAuth client id / scope answers under project-standard UPPER_SNAKE names so that the oauth/register step above can resolve them via ${{...}} references at provision time. The MCP_DA_AUTH_ID_APIGITHUBC= entry stays empty — oauth/register writes the resolved configuration id back into it (env-var suffix is derived from the MCP server hostname, e.g. api.githubcopilot.com → APIGITHUBC). The client secret never lands here: it goes to env/.env.dev.user below, which is gitignored and auto-masked in logs.

env/.env.dev — OAuth (static) shown
1# Built-in environment variables
2TEAMSFX_ENV=dev
3APP_NAME_SUFFIX=dev
4TEAMS_APP_ID=
5MCP_DA_AUTH_ID_APIGITHUBC=
6MCP_DA_OAUTH_CLIENT_ID_APIGITHUBC=<value entered at create>
7MCP_DA_OAUTH_SCOPE_APIGITHUBC=<value entered at create>
env/.env.dev.user gitignored, auto-masked in logs
1SECRET_MCP_DA_OAUTH_CLIENT_SECRET_APIGITHUBC=<value entered at create>

Entra SSO writes only MCP_DA_OAUTH_CLIENT_ID_APIGITHUBC — the driver suppresses secret and scope when identityProvider === MicrosoftEntra. OAuth (dynamic) writes neither — dcr/register obtains the credentials from the authorization server at provision. None writes no auth entries at all.

.vscode/mcp.json generated (all auth types)

Written to disk but not auto-opened after scaffold — the create flow ends on the success notification, not on this file. The developer opens it manually to exercise the MCP server from Copilot Chat, or as a fallback path: if dynamic tool discovery in ai-plugin.json doesn’t work for the chosen server, the ⚡ ATK: Fetch action from MCP CodeLens bakes the discovered tool definitions into the action manifest.
