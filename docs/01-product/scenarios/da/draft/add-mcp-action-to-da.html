Visualizes the redesigned VS Code Add action flow from add-mcp-action-to-da.md (draft). Feature flag

This redesign is gated behind the same two flags that scope the companion create scenario — both default off so that current shipping behavior is unchanged.

TEAMSFX_MCP_FOR_DA_DT default: false

Gates the VS Code inline add-action flow documented on this page. When the flag is on, the Add action command runs the full URL → auth-type → follow-up → write-files flow inline. When the flag is off, VS Code falls back to shipped behavior: a .vscode/mcp.json entry is written and the developer collects auth later via the fetch-mcp-tools CodeLens (SCN-DA-FETCH-MCP-TOOLS).

The same flag also gates the credential-persistence subset: UPPER_SNAKE env entries (MCP_DA_OAUTH_CLIENT_ID_* / SECRET_MCP_DA_OAUTH_CLIENT_SECRET_* / MCP_DA_OAUTH_SCOPE_*) and the matching ${{...}} references in m365agents.yml are written only when this flag is on.

CLI tools-file requirement: when DT is off, --mcp-tools-file-path is required (shipped behavior — static tools list). When DT is on, --mcp-tools-file-path is optional; omitting it routes to dynamic discovery and the generated ai-plugin.json runtime carries enable_dynamic_discovery: true.

TEAMSFX_MCP_FOR_DA_DCR default: false

Gates only the OAuth (with dynamic client registration) auth-type option in step 3 and the matching dcr/register action in m365agents.yml. The option is shown — and the CLI value oauth-dynamic is accepted — only when both TEAMSFX_MCP_FOR_DA_DT and TEAMSFX_MCP_FOR_DA_DCR are true, because DCR scaffolding currently piggybacks on the DT-mode env-file plumbing. Default off; temporary — removed once DCR ships unconditionally.

VS Code Add Action Flow

Runs the Add action command against an existing DA project. The toolkit writes the action manifest, updates the DA manifest, and updates m365agents.yml in one shot. No separate fetch step.

1. Add an Action: pick action type

2. Enter MCP server URL

3. Select authentication type new

VS Code gate: this pick (and follow-up cards 3a/3b) is only shown when TEAMSFX_MCP_FOR_DA_DT is true. When the flag is off, the Add action command instead writes a .vscode/mcp.json entry and the developer collects auth later via the fetch-mcp-tools CodeLens (SCN-DA-FETCH-MCP-TOOLS) — matching shipped behavior. CLI is unaffected.

Note: the OAuth (with dynamic client registration) option is only shown when both TEAMSFX_MCP_FOR_DA_DT and TEAMSFX_MCP_FOR_DA_DCR are true (see Feature flag section). With either flag off, the pick has three options.

3a. OAuth (static) follow-up only when option above is selected

3b. Entra SSO follow-up only when Entra SSO is selected

No manifest picker, no confirmation modal core.addPlugin VS Code + MCP + DT-on branch

VS Code does not ask for a Teams manifest.json path — core.addPlugin infers it from projectPath/appPackage/manifest.json when the selectTeamsAppManifestQuestion is skipped for VS Code + MCP. It also does not show a core.addApi.confirm.mcp / .mcp.teamsYaml warning modal before writing — once the last required answer (auth follow-up field if any, otherwise the auth-type pick) is provided, the flow writes the action manifest, updates the DA manifest, and updates m365agents.yml immediately, then shows the Action added info notification (step 4) with a follow-up Provision button. The CLI interactive flow still goes through core.addPlugin and keeps the warning modal; CLI non-interactive suppresses it with --interactive false.

Files written after success

Shown for an MCP action added to an existing DA (which already contains action from the create flow). Each new MCP action gets an auto-incremented _N suffix on its id and file name to keep them distinct; here the new action is named action_2 against https://api.githubcopilot.com/mcp/, with authentication type OAuth (with static registration). The oauth/register step is injected by ActionInjector.injectCreateOAuthActionForMCP in m365agents.yml for OAuth (static) and Entra SSO; the static variant adds authorizationUrl/tokenUrl/refreshUrl with identityProvider: Custom, the SSO variant uses identityProvider: MicrosoftEntra only. OAuth (dynamic) instead injects a dcr/register step (DCR — Dynamic Client Registration, RFC 7591). None writes no auth step and omits the manifest auth block. Env-var suffixes are derived from the MCP server hostname (here api.githubcopilot.com → APIGITHUBC), matching the create scenario; the same MCP server reused across actions shares the same auth-id placeholder. The credential-trio env entries (MCP_DA_OAUTH_CLIENT_ID_* / SECRET_MCP_DA_OAUTH_CLIENT_SECRET_* / MCP_DA_OAUTH_SCOPE_*) and the corresponding ${{...}} references in m365agents.yml are written only when feature flag TEAMSFX_MCP_FOR_DA_DT is on; otherwise env/.env.<env> contains just the auth-id placeholder and provision picks credentials up through the in-process bridge.

appPackage/ai-plugin_2.json new

The new runtimes entry carries an auth block: OAuthPluginVault with a ${{MCP_DA_AUTH_ID_*}} reference_id for static OAuth / Entra SSO / OAuth dynamic, or { "type": "None" } for the None auth choice. Matches FxCore.declarativeAgent.updateActionWithMCP (CodeLens path) and the inline branch of core.addPlugin (VS Code Add action) — both derive auth via deriveMCPManifestOAuth(authType, registrationId) ?? { type: "None" }.

m365agents.yml + auth registration step

Injected immediately after the existing teamsApp/create step. Same shapes as the create scenario; only the action name differs (here action_2). Per auth type:

  • OAuth (static) — oauth/register with identityProvider: Custom plus authorizationUrl / tokenUrl / refreshUrl, and (when TEAMSFX_MCP_FOR_DA_DT is on) clientId / clientSecret / scope as ${{...}} references to the UPPER_SNAKE env entries the add-action flow persisted. Injected by ActionInjector.injectCreateOAuthActionForMCP.
  • Entra SSO — same oauth/register shape with identityProvider: MicrosoftEntra, no URL trio, and only clientId: ${{MCP_DA_OAUTH_CLIENT_ID_APIGITHUBC}} (driver suppresses secret / scope / confirm).
  • OAuth (dynamic) — dcr/register (RFC 7591 Dynamic Client Registration) with name: action_2, appId: ${{TEAMS_APP_ID}}, applicableToApps: SpecificApp, targetAudience: HomeTenant, wellKnownAuthorizationServer: <RFC 8414 metadata URL>, targetUrlsShouldStartWith: [ <mcp url> ], and the same writeToEnvironmentFile.configurationId: MCP_DA_AUTH_ID_APIGITHUBC. No client-credentials prompts — the authorization server issues them dynamically at provision.
  • None — no registration step.

env/.env.dev add-action persists OAuth answers; configurationId placeholder filled at provision

When TEAMSFX_MCP_FOR_DA_DT is on, the add-action flow writes the OAuth client id / scope answers under project-standard UPPER_SNAKE names so that the oauth/register step above can resolve them via ${{...}} references at provision time. The MCP_DA_AUTH_ID_APIGITHUBC= entry stays empty — oauth/register writes the resolved configuration id back into it (env-var suffix is derived from the MCP server hostname, e.g. api.githubcopilot.com → APIGITHUBC). The client secret never lands here: it goes to env/.env.dev.user below, which is gitignored and auto-masked in logs. With the flag off, only the auth-id placeholder is appended and the credential trio is omitted.

env/.env.dev — OAuth (static) shown
1# Built-in environment variables
2TEAMSFX_ENV=dev
3APP_NAME_SUFFIX=dev
4TEAMS_APP_ID=
5MCP_DA_AUTH_ID_APIGITHUBC=
6MCP_DA_OAUTH_CLIENT_ID_APIGITHUBC=<value entered at add-action>
7MCP_DA_OAUTH_SCOPE_APIGITHUBC=<value entered at add-action>
env/.env.dev.user gitignored, auto-masked in logs
1SECRET_MCP_DA_OAUTH_CLIENT_SECRET_APIGITHUBC=<value entered at add-action>

Entra SSO writes only MCP_DA_OAUTH_CLIENT_ID_APIGITHUBC — the driver suppresses secret and scope when identityProvider === MicrosoftEntra. OAuth (dynamic) writes neither — dcr/register obtains the credentials from the authorization server at provision. None writes no auth entries at all.
